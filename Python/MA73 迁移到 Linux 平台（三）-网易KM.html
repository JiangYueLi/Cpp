将 MA73 进程从运行在 Windows 平台的 C++ MFC Server 迁移到 Linux 平台上的进程， 本文章是系列第三篇

前情提要

由于迁移跨度比较大， 所以打算分成三步走

  1. 先在 Window 平台上将 MFC 服务器变成 Console 服务器（去除窗口相关内容）

  2. 改造直接使用 Window 平台功能的代码， 使用跨平台代码进行替换（尽量能同名替换， 这样可以减少其他相关代码修改）

  3. -> 根据 Linux 平台调整第三方库以及原先相关功能

准备编译和运行环境， docker

为了减少编译和链接期间 symbol 的重复定义现象， 所以手动编译的第三方库均采用 so 方式（虽然都会编译出对应的 .a 版本）

 

boost

准备的版本是 1.72.0 跟 Windows 使用的版本一直， 编译出两个版本，

  • debug 版本 安装在 /usr/local/debug/

  • release 版本安装在 /usr/local/

编译流程参考： https://www.boost.org/doc/libs/1_72_0/more/getting_started/unix-variants.html

 

cpprestsdk

准备的版本是 2.10.18

编译参数如下

# debug
cmake -G Ninja -DCMAKE_BUILD_TYPE=Debug -DBUILD_SHARED_LIBS=1 -DCMAKE_INSTALL_PREFIX=/usr/local/debug
ninja
ninja install

# release
cmake -G Ninja -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=1 -DCMAKE_INSTALL_PREFIX=/usr/local
ninja
ninja install

 

mysql connector c++

使用的版本 8.0.31， 编译分 debug/release 两个版本

debug 编译参数

cmake -DWITH_JDBC=OFF -DCMAKE_C_COMPILER=gcc -DCMAKE_CXX_COMPILER=g++ -DCMAKE_BUILD_TYPE=debug -DCMAKE_INSTALL_PREFIX=/usr/local/debug -DWITH_BOOST=/usr/local/include -DWITH_SSL=system -DBUILD_STATIC=OFF -DWITH_ZLIB=system -DWITH_LZ4=system -DWITH_PROTOBUF=system
cmake --build . --config Debug
cmake --build . --target install --config Debug

release 编译参数

cmake -DWITH_JDBC=OFF -DCMAKE_C_COMPILER=gcc -DCMAKE_CXX_COMPILER=g++ -DCMAKE_BUILD_TYPE=release -DCMAKE_INSTALL_PREFIX=/usr/local -DWITH_BOOST=/usr/local/include -DWITH_SSL=system -DBUILD_STATIC=OFF -DWITH_ZLIB=system -DWITH_LZ4=system -DWITH_PROTOBUF=system
cmake --build . --config Release
cmake --build . --target install --config Release

至于为啥 WITH_JDBC=OFF 原因如下

  • 当前新写的代码只使用 mysqlx dev api , 没有使用 jdbc

  • jdbc 在 8.0 之后已经标记为弃用

  • jdbc 编译时需要安装 mysql server 相关内容， 比较麻烦

编译流程参考： https://dev.mysql.com/doc/connector-cpp/8.0/en/connector-cpp-apps-general-considerations.html#connector-cpp-apps-version-macros

编写 cmake

cmake 结构， source 目录下 CMakeList.txt 作为主入口， 然后各个工程目录下都添加 CMakeList.txt.

工程名： ServerCommon, NetEngine, ServerWorld, ServerLogin, ServerChatting, ServerGame

编译的话， 可以在 source 目录下编译全部， 也可以在上面工程目录里单独编译

编译参数

  • -DUSE_ASAN=ON 开启 AddressSanitizer, 用于内存检测， 主要检测内存 heap-use-after-free use-after-free.

  • 相关内容参考: https://github.com/google/sanitizers/wiki/AddressSanitizer

  • -DUSE_TCMALLOC=ON 设置 tcmalloc 为默认 malloc, 当前默认已经开启

  • -DUSE_PROFILER=ON 开启 cpu profiler 功能

  • 相关内容参考： https://github.com/gperftools/gperftools

编译脚本

# 开启 AddressSanitizer 和 cpu profiler 功能
cmake -DUSE_ASAN=ON -DUSE_TCMALLOC=ON -DUSE_PROFILER=ON .
make -j8

class PUBLIC_API Value
  : public virtual Printable
{
public:

  enum Type
  {
    VNULL,      ///< Null value
    UINT64,     ///< Unsigned integer
    INT64,      ///< Signed integer
    FLOAT,      ///< Float number
    DOUBLE,     ///< Double number
    BOOL,       ///< Boolean
    STRING,     ///< String (utf8)
    USTRING,    ///< Wide string (utf16)
    RAW,        ///< Raw bytes
    EXPR,       ///< String to be interpreted as an expression
    JSON,       ///< JSON string
  };

  using string = std::string;

protected:

  Type m_type;

  // TODO: Use std::variant to save space

  DLL_WARNINGS_PUSH

  std::string     m_str;
  std::u16string  m_ustr;

  DLL_WARNINGS_POP

  union {
    double   v_double = 0.0;
    float    v_float;
    int64_t  v_sint;
    uint64_t v_uint;
    bool     v_bool;
  } m_val;

  //...

  bool is_null() const
  {
    return VNULL == m_type;
  }

  bool get_bool() const
  {
    switch (m_type)
    {
    case BOOL:   return m_val.v_bool;
    case UINT64: return 0 != m_val.v_uint;
    case INT64:  return 0 != m_val.v_sint;
    case FLOAT:  return 0.0f != m_val.v_float;
    case DOUBLE: return 0.0 != m_val.v_double;
    case VNULL:  return false;
    default:
      throw Error("Can not convert to Boolean value");
    }
  }

  uint64_t get_uint() const
  {
    // 添加支持 null, float, double to int
    if (UINT64 != m_type && INT64 != m_type && BOOL != m_type && VNULL != m_type && FLOAT != m_type && DOUBLE != m_type)
      throw Error("Can not convert to integer value");
    
    // 应对建表的时候设置默认为 NULL
    if (is_null())
        return 0;

    // 针对浮点数
    if (FLOAT == m_type)
        return (uint64_t)m_val.v_float;
    if (DOUBLE == m_type)
        return (uint64_t)m_val.v_double;

    if (BOOL == m_type)
      return m_val.v_bool ? 1 : 0;

    if (INT64 == m_type && 0 > m_val.v_sint)
      throw Error("Converting negative integer to unsigned value");

    uint64_t val = (UINT64 == m_type ? m_val.v_uint : (uint64_t)m_val.v_sint);

    return val;
  }

  int64_t get_sint() const
  {
    if (INT64 == m_type)
      return m_val.v_sint;

    uint64_t val = get_uint();

    if (!check_num_limits<int64_t>(val))
      throw Error("Value cannot be converted to signed integer number");

    return val;
  }

  float get_float() const
  {
    switch (m_type)
    {
    case INT64:  return 1.0F*m_val.v_sint;
    case UINT64: return 1.0F*m_val.v_uint;
    case FLOAT:  return m_val.v_float;
    case DOUBLE: return (float)m_val.v_double;
    case BOOL: return m_val.v_bool ? 1.0f : 0.0f;
    case VNULL: return 0.0f;
    default:
      throw Error("Value cannot be converted to float number");
    }
  }

  double get_double() const
  {
    switch (m_type)
    {
    case INT64:  return 1.0*m_val.v_sint;
    case UINT64: return 1.0*m_val.v_uint;
    case FLOAT:  return m_val.v_float;
    case DOUBLE: return m_val.v_double;
    case BOOL: return m_val.v_bool ? 1.0 : 0.0;
    case VNULL: return 0.0;
    default:
      throw Error("Value can not be converted to double number");
    }
  }

  /*
    Note: In general this method returns raw value representation as obtained
    from the server, which is stored in m_str member. If a non-string value was
    not obtained from the server, there is no raw representation for it and
    error is thrown. String values always have raw representation which is
    either utf8 or utf16 encoding. Strings obtained from the server use utf8 as
    raw representation. For strings created by user code this might be either
    utf8 or utf16, depending on how string was created.
  */

  const byte* get_bytes(size_t *size) const
  {
    switch (m_type)
    {
    case USTRING:
      if (!m_ustr.empty())
      {
        if (size)
          *size = m_ustr.size() * sizeof(char16_t);
        return (const byte*)m_ustr.data();
      }
      FALLTHROUGH;

    default:
      if (m_str.empty())
        throw Error("Value cannot be converted to raw bytes");
      FALLTHROUGH;

    case RAW:
    case STRING:
      if (size)
        *size = m_str.length();
      return (const byte*)m_str.data();
    case VNULL:
        if (size)
            *size = 0;
        return nullptr;
    }
  }
// ...
};

 

  • 特别处理 mysql datetime 类型返回值内容， 由于 datetime 返回值会有两种类型情况， 一种是 string 类型， 另一种则是 RAW 类型

  • string 类型处理起来比较简单， 主要是根据 年-月-日 时:分:秒 或者 时:分:秒 格式处理即可

  • RAW 类型比较复杂， 这个类型返回的结果是 protobuf 原生数据， 对数据进行 protobuf 不定长数字解析

  • 还需要注意的是由于 raw 类型发送的内容不一定完全比如说 mysql 可能就只发送了 年， 月， 日， 所以需要自行补足相关内容

  • protobuf 不定长数字解析说明： https://developers.google.com/protocol-buffers/docs/encoding#varints

static inline std::vector<uint64_t> mysqlx_raw_as_u64_vector(const mysqlx::Value& in_value) {
	std::vector<uint64_t> out;

	const auto bytes = in_value.getRawBytes();
	auto ptr = reinterpret_cast<const unsigned char*>(bytes.first);
	auto end = reinterpret_cast<const unsigned char*>(bytes.first) + bytes.second;

	while (ptr != end) {
		static constexpr unsigned char carry_flag{ 0b1000'0000 };
		static constexpr unsigned char value_mask{ 0b0111'1111 };

		uint64_t v = 0;
		uint64_t shift = 0;
		bool is_carry;
		do {
			auto byte = *ptr;
			is_carry = (byte & carry_flag) == carry_flag;
			v |= (uint64_t)(byte & value_mask) << shift;

			++ptr;
			shift += 7;
		} while (is_carry && ptr != end && shift <= 63);

		out.push_back(v);
	}

	return out;
}

配置表字符串和协议字符串

由于在 windows 上 wchar_t 长度为 2byte, 也就是 UTF-16, 而在 Linux 上 wchar_t 长度为 4byte， 也就是 UTF-32, 所以涉及的相关代码需要做兼容， 主要涉及到配置表和协议需要调整

  • 配表： 由于配置表的字符串是使用 C# 进行输出到文件里。 由于 C# 字符串默认是 UTF-16， 但是由于 wchar_t 在 Linux 平台是 UTF-32 所以需要进行转化。 做法是先将配置文件里的内容读出之后， 先转化 UTF-8 即 char, 然后调用先前的 char 转化为 wchar_t 功能转化为 utf-32 供代码其他代码使用

  • 协议：协议也是类型， 由于先前客户端和服务端通讯的时候是使用 windows wchar_t 字符串即 UTF-16, 所以每次传输的协议打包和解析都需要额外做处理。

  • 打包： 先将 wstring 从 utf-32 转化成 utf-8, 然后调用代码将 utf-8 转化为 utf-16， 相关代码文件为 CommonSource/ProtocolUtil/Serializer.hpp

  • 解析： 先将收到协议， 从 UTF-16 转化为 UTF-8, 然后再将 UTF-8 转化为 UTF-32

 

其他

  • 泛型特例化： 这个情况主要出现在协议文件和 memorypool 相关功能， 在泛型特例化时， 需要在函数开头添加 template <>, 但是在 windows 平台上不加也不会报错， 但是在 Linux 平台就会报错。

  • 协议文件， 所有文件名中带有 ProtocolHandler 的都是。 这个处理没有别的办法就是找出所有的协议文件进行修改

  • 涉及到的函数只有一个就是 TProtocolHandler

  • memorypool 这块的话， 直接去除 memorypool 相关功能， 直接将改写的 operator new/delete 使用 malloc/free 进行处理， 外部直接接入 tcmalloc 来优化。

cpprestsdk 相关的挫折

由于 cpprestsdk 在 windows 平台上处理 json 解析时是以 UTF-16 形式进行的（即 wstring）， 对于 string 类型做了兼容。 但是在 Linux 平台上则是以 UTF-8 为基准，且没有对 wstring 类型进行兼容。 但是项目代码里大量使用 wstring 进行 json 查询或者使用。 对于在 json 使用 wstring 作为参数的函数兼容比较简单即只需要添加重载函数即可实现， 对于返回值发送变动的情况就相当麻烦。 有两个函数属于这类问题： as_string 和 serialize, 这两个函数在 Linux 平台上正常情况下是返回 string 而非 wstring, 但是 73 工程里要求的返回值都是 wstring.

为了解决这类问题， 现在 json.h 文件里定义 wstring 版本的相关函数, as_string -> as_wstring, serialize -> serializeW

然后在头文件结尾添加如下宏定义

#ifdef CPPRESTSDK_COMPATIBLE
#define as_string as_wstring
#define serialize serializeW
#endif

 

然后为了防止 json.h 多次被 include 导致影响自身，则在开头添加如下代码

#ifdef CPPRESTSDK_COMPATIBLE
    #ifdef as_string
    #undef as_string
    #endif
    #ifdef serialize
    #undef serialize
    #endif
#endif

这样之后， 但凡包含了 json.h 的文件， 该文件所有的 as_string 都会被宏定义替换为 as_wstring, serialize 同理

最后只需要在 CMakeList.txt 里添加 CPPRESTSDK_COMPATIBLE 即可
